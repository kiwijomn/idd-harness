# harness-brainstorm

아이디어를 한 번에 다 펼치지 않는다.
**한 라운드에 하나의 구체적 출력 + 하나의 확인 질문**으로 점진적으로 좁혀간다.

목표: 막연한 아이디어 → 하나의 작은 Issue로 만들 수 있을 만큼 구체적인 계획

## 진행 방식

### Round 1: 핵심 파악
사용자의 말에서 가장 중요한 가정 하나를 찾아 표면화한다.

```
[현재 이해]: {한 문장으로 요약}
[가정]: {숨겨진 가정 하나}
[질문]: {이 가정이 맞는지 확인하는 질문 하나}
```

### Round 2+: 범위 좁히기
피드백을 받을 때마다 하나씩 구체화한다. 여러 방향을 동시에 펼치지 않는다.

- 선택지가 2개 이상이면 → 추천 하나 + 이유 제시, 사용자가 고름
- 불명확한 부분이 있으면 → 그것부터 해결, 그 다음 진행

### 종료 조건
다음이 모두 명확해지면 brainstorm 종료 가능:

- [ ] 만들 것이 한 문장으로 정의됨
- [ ] 성공 기준이 검증 가능한 형태로 존재
- [ ] 범위가 하나의 Issue로 만들기에 충분히 작음 (한 명이 하루 안에 구현 가능한 크기)

**종료 전 필수**: 위 3가지를 사용자에게 보여주고 확인을 받는다. Claude가 단독으로 종료를 결정하지 않는다.

```
[brainstorm 종료 제안]
✅ 만들 것: {한 문장}
✅ 성공 기준: {검증 가능한 형태}
✅ 범위: {Issue 하나로 만들기에 충분}

harness-plan으로 넘어갈까요?
```

사용자가 "아직 아니야" 또는 수정 요청을 하면 → Round N+1 계속 진행

종료 확인 후 → `harness-plan` 호출

## karpathy §1 내장 원칙

- 불명확한 상태에서 설계하지 않는다.
- 여러 해석이 있으면 하나를 선택하지 않고, 모두 보여주고 선택하게 한다.
- 더 단순한 방법이 있으면 먼저 제안한다.

## 금지 사항

- 한 번에 5가지 방향 제시 금지
- 사용자 확인 없이 다음 단계로 진행 금지
- "일단 다 만들고 나중에 좁히자" 방식 금지

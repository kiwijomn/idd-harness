# harness-debug

에러가 나면 고치기 전에 먼저 이해한다.
**같은 명령어를 다시 실행하지 않는다. 같은 방법을 두 번 시도하지 않는다.**

## 진단 우선 (karpathy §1)

실행 전에 답해야 하는 질문:
```
1. 정확히 어떤 에러인가? (메시지 전문 확인)
2. 언제부터 발생했는가? (마지막으로 동작한 시점)
3. 무엇이 변경되었는가? (git diff, 최근 커밋)
4. 재현 가능한가? (재현 조건)
```

이 질문에 답하기 전에 코드를 수정하지 않는다.

## 디버깅 루프

```
[관찰]: 에러 메시지 / 스택 트레이스 전문 확인
    ↓
[가설]: 원인 후보 1~2개 (3개 이상이면 범위가 너무 넓음)
    ↓
[검증]: 가설을 검증하는 최소한의 테스트 또는 로그
    ↓
[수정]: 원인이 확인된 경우에만 코드 수정 (karpathy §3 Surgical)
    ↓
[verify]: 재현 케이스가 통과하는지 확인
```

## karpathy §5 실패 분석

- 같은 접근으로 두 번 실패하면 → 접근 자체를 버린다
- 증상만 고치면 근본 원인이 남는다 → 근본 원인을 찾는다
- 작동하는 것처럼 보이는 것과 실제로 작동하는 것을 구분한다

## 재현 테스트 먼저

버그를 고치기 전에 버그를 재현하는 테스트를 먼저 작성한다.:
```
실패하는 테스트 작성 → 테스트 통과하게 수정 → 커밋
```
이 순서가 뒤집히면 나중에 같은 버그가 다시 나타난다.

## 디버그 완료 후

원인과 수정 내용이 명확해지면 → `harness-execute`로 복귀
